    Instrumentation Deep-Dive — Code Walk-through¶

    Presenter script for a live code walk-through of how Spaceport instruments three services (React, Go, Python) using OpenTelemetry and Weaver-generated semantic conventions. Start in the editor, optionally finish in Grafana.

    0. The Architecture (30 sec)¶

    Browser (React)  →  API (Go/Gin)  →  Pricing Service (Python/FastAPI)
       ↓                 ↓                     ↓
     OTLP             OTLP                  OTLP
       └────────── OTel Collector ──────────┘
                   ↓       ↓       ↓
                Tempo  Prometheus  Loki
                       ↓
                    Grafana

    Three services, three languages — same telemetry contract, zero hardcoded strings. Let's trace how that works, starting from the content layer.

    1. The Content Layer — No Telemetry Here¶

    Open frontend/src/data/informationDesk.ts.

    Line What to show Talking point
    1–10 File header Content registry — edit this array to change what the app shows. No component changes needed
    13–50 InfoExhibit interface The shape of each exhibit: id, title, intro, points, image. Pure data
    55–73 pageConfig Page-level config. The *asterisks* in title highlight a word. Content drives the UI
    80–85 Start of exhibits[] Each exhibit is a self-contained record. Rendering adapts automatically

    Transition: "This is just content. It has no idea it's being observed. Now let's see where we define what to observe."

    2. The Semantic Convention Registry (Source of Truth)¶

    Key message: All attribute names, metric definitions, and span schemas are defined once in YAML. Every service generates code from these files.

    2a. Attributes¶

    Open semconv/models/attributes/exhibit.yaml.

    Line What to show Talking point
    1–6 Group header registry.spaceport.exhibit This YAML file is the single source of truth for any attribute prefixed spaceport.exhibit.*
    7–13 spaceport.exhibit.id Type=string, stability, brief, examples — all captured as structured data
    14–19 spaceport.exhibit.title Human-readable title. Same name will appear in Go, Python, TypeScript
    26–30 spaceport.exhibit.count Used by the page-level span to record how many exhibits the page has

    2b. Span Definitions¶

    Open semconv/models/spans/frontend.yaml.

    Line What to show Talking point
    1–10 span.spaceport.user.view_information_desk type: span, span_kind: internal, required attr spaceport.exhibit.count. This defines the contract
    12–26 event.spaceport.exhibit_viewed An event definition — fired when an exhibit scrolls into view. Required exhibit.id, recommended title + number
    56–66 span.spaceport.exhibit.view Separate span per exhibit — linked (not child) to the page span
    68–75 span.spaceport.app.load_currencies Startup span for loading currencies
    77–84 span.spaceport.user.change_currency User interaction span — required attrs: base_currency, display_currency

    2c. Metric Definitions¶

    Open semconv/models/metrics/frontend.yaml.

    Line What to show Talking point
    1–16 spaceport.frontend.page_views Counter, unit={view}, required attr page.name. Every page fires the same counter
    18–31 spaceport.frontend.exhibit_views Counter per exhibit. Tracks scroll engagement
    33–42 spaceport.frontend.exhibit_dwell_time Histogram, unit=seconds. How long each exhibit was visible — p50/p95 in Grafana
    44–60 spaceport.frontend.bookings Booking attempts. outcome=success/failure, seat.class=optional dimension

    Transition: "From these YAML files, Weaver generates type-safe code. One command: make generate."

    3. Code Generation: YAML → Three Languages¶

    Show that one YAML definition produces constants in TS, Go, and Python.

    TypeScript (generated)¶

    Open frontend/src/semconv/attribute.ts — scroll to the SPACEPORT_EXHIBIT_* block.

    What to show Talking point
    Header: "DO NOT MODIFY: autogenerated" Developers never hand-edit these
    export const SPACEPORT_EXHIBIT_ID = "spaceport.exhibit.id" Typed constant — typo-proof, grep-friendly, IDE-autocompleted

    Open frontend/src/semconv/metric.ts — search SpaceportFrontendExhibitDwellTime.

    What to show Talking point
    class SpaceportFrontendExhibitDwellTime Generated class wrapping an OTel Histogram
    record(value, spaceport_exhibit_id) method You cannot forget the required attribute — it's a function parameter

    Open frontend/src/semconv/span.ts — search startSpaceportUserPlaceBooking.

    What to show Talking point
    SPAN_SPACEPORT_USER_PLACE_BOOKING_NAME constant Same name in every language
    function startSpaceportUserPlaceBooking(tracer, departure_id, seat_class) Required attributes enforced at compile time. Span kind set automatically

    Go (generated)¶

    Open api/internal/semconv/span.go — search StartSpaceportDepartureListServer.

    What to show Talking point
    const SpanSpaceportDepartureListServerName Same value as the TypeScript and Python constants
    func StartSpaceportDepartureListServer(ctx, tracer, opts...) Sets SpanKindServer — you can't accidentally create a CLIENT span

    Python (generated)¶

    Open pricing-service/pricing_service/semconv/span.py — search start_spaceport_pricing_calculate_server.

    What to show Talking point
    SPAN_SPACEPORT_PRICING_CALCULATE_SERVER_NAME Exact same string value as Go and TypeScript
    def start_spaceport_pricing_calculate_server(tracer, spaceport_departure_id, spaceport_seat_class) start_as_current_span under the hood — works as a Python context manager

    4. Frontend Instrumentation (TypeScript)¶

    4a. OTel Bootstrap¶

    Open frontend/src/instrumentation.ts.

    Line What to show Talking point
    1–21 Imports: SDK + generated metric classes The app imports SpaceportFrontendPageViews, SpaceportFrontendBookings, etc. from generated code — not raw strings
    26–30 resource creation Standard OTel resource: service.name, service.version, deployment.environment.name
    33–41 Trace provider BatchSpanProcessor → OTLP exporter at /otlp/v1/traces. Vite dev server proxies this to the Collector
    56–58 visibilitychange handler forceFlush() on tab hide. Without this, spans are lost when users close the tab
    80–81 tracer + meter exports Single instances shared by all pages
    84–87 Metric instantiation new SpaceportFrontendPageViews(meter) — generated typed class, created once, imported everywhere

    4b. The useSpan Hook¶

    Open frontend/src/hooks/useSpan.ts.

    Line What to show Talking point
    12–16 useSpan(name, attributes?) Reusable React hook. Starts a span on mount, returns spanRef and endSpan()
    22–26 useEffect → tracer.startSpan(name, ...) Span starts when the component mounts
    29–31 Cleanup function: span.end() Safety net on unmount — the span always ends
    37–41 endSpan() callback Call after data loads. Gives the span a meaningful duration instead of "mount → unmount"

    4c. Information Desk — All Three Signals¶

    Open frontend/src/pages/InformationDeskPage.tsx.

    Line What to show Talking point
    6–13 Imports from instrumentation + generated code Everything typed. pageViewCounter, exhibitViewCounter, exhibitDwellHistogram
    19–22 Span name imports SPAN_SPACEPORT_USER_VIEW_INFORMATION_DESK_NAME, SPAN_SPACEPORT_EXHIBIT_VIEW_NAME
    33–35 useSpan(...) with exhibit count Trace: page-level span. spaceport.exhibit.count = how many exhibits exist
    43–47 pageViewCounter.add(1, "information_desk") + logger.emit(...) Metric + Log on mount. One user action → three signals
    82 IntersectionObserver rootMargin: "-45% 0px" — only a ~10% vertical center strip counts as "visible"
    95–107 Exhibit enters viewport tracer.startSpan(SPAN_SPACEPORT_EXHIBIT_VIEW_NAME, { links: [pageSpanCtx] }) — linked span, not a child. Shows "related to" without nesting
    109–118 First view of exhibit Counter bump + span event + structured log — all correlated by trace context
    120–128 Exhibit leaves viewport Span ends. Dwell histogram records seconds. In Grafana → p50/p95 per exhibit

    Demo tip: Scroll the Information Desk in the browser. Then show Tempo — you'll see the page span with linked exhibit spans fanning out.

    4d. Booking — Cross-Service Trace Start¶

    Open frontend/src/pages/BookingFormPage.tsx.

    Line What to show Talking point

    72 startSpaceportUserChangeCurrency(tracer, "UNC", code) User interaction span with both currencies as required attrs

    5b. Departure Handler — Server Span¶

    Open api/internal/handler/departures.go.

    Line What to show Talking point
    20 var tracer = otel.Tracer("spaceport-api") One tracer per package
    50–51 semconv.StartSpaceportDepartureListServer(ctx, tracer) Generated function → correct span name + SpanKindServer

    5c. Booking Handler — Metrics + Error Path¶

    Open api/internal/handler/bookings.go.

    Line What to show Talking point
    44 tracer.Start(ctx, semconv.SpanSpaceportBookingCreateServerName) Span name from generated constant. Attributes added later (booking_id not known yet)
    53–55 span.SetAttributes(semconv.AttrSpaceportDepartureIdKey.Int(...)) .Int() for integer departure ID — type safety from generated code
    60 span.SetAttributes(semconv.AttrSpaceportBookingId(bookingID)) Booking ID added after UUID generation

    7. The Full Trace (Live in Grafana)¶

    Make a booking in the browser, then open the trace in Tempo.

    Expected trace waterfall:

    [frontend]  user.place_booking          (INTERNAL)
  └─ [api]  booking.create.server       (SERVER)
       ├─ [api]  pricing.calculate.client   (CLIENT)
       │    └─ [pricing] pricing.calculate.server  (SERVER)
       └─ booking_completed             (EVENT)

    Point out in the trace:

    1. Span kinds — INTERNAL → SERVER → CLIENT → SERVER. The CLIENT/SERVER pair creates the service boundary
    2. Attributes — spaceport.booking.id, spaceport.seat.class, spaceport.departure.id visible on relevant spans
    3. Events — booking_completed with booking ID
    4. Three services, one trace — traceparent header connects them

    9. Closing¶

    "One YAML registry. Three languages. Zero hardcoded telemetry strings. When the schema changes, make generate updates every service. That's observability-by-design."
