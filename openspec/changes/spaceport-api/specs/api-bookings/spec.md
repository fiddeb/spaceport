## ADDED Requirements

### Requirement: POST /api/bookings delegates to pricing and persists
Covered in api-departures spec. This capability spec covers the booking state and lifecycle aspects: status transitions, error persistence, and response contract.

### Requirement: Booking status persisted for all outcomes
Both successful and failed bookings SHALL be persisted to the `bookings` table in SQLite. Failed bookings SHALL have `status = "failed"` and `error_reason` populated.

#### Scenario: Failed booking is stored
- **WHEN** `POST /api/bookings` fails because the pricing service returns an error
- **THEN** a booking record with `status = "failed"` is written to SQLite and the error_reason is logged

### Requirement: Booking ID is a UUID
Each booking SHALL be assigned a UUID v4 as `booking_id` generated by the API (not the database).

#### Scenario: Booking response contains UUID
- **WHEN** a booking is successfully created
- **THEN** the `booking_id` in the response is a valid UUID v4 format string

### Requirement: In-flight booking uses active trace context
When processing a booking, the API SHALL pass the active trace context through all function calls (SQLite write, pricing HTTP call) so that all child spans appear under the same trace.

#### Scenario: All booking sub-operations share trace
- **WHEN** a booking is processed end-to-end
- **THEN** Tempo shows frontend, API server, `api.process_booking`, `api.call_pricing_service`, and SQLite spans all under the same `trace_id`
