# Instrumentation Deep-Dive — Code Walk-through

> Presenter script for a live code walk-through of how Spaceport instruments
> three services (React, Go, Python) using OpenTelemetry and Weaver-generated
> semantic conventions. Start in the editor, optionally finish in Grafana.

---

## 0. The Architecture (30 sec)

```
Browser (React)  →  API (Go/Gin)  →  Pricing Service (Python/FastAPI)
       ↓                 ↓                     ↓
     OTLP             OTLP                  OTLP
       └────────── OTel Collector ──────────┘
                   ↓       ↓       ↓
                Tempo  Prometheus  Loki
                       ↓
                    Grafana
```

Three services, three languages — same telemetry contract, zero hardcoded
strings. Let's trace how that works, starting from the content layer.

## 4. Frontend Instrumentation (TypeScript)

### 4a. OTel Bootstrap

Open **frontend/src/instrumentation.ts**.

| 1–21 | Imports: SDK + generated metric classes | The app imports `SpaceportFrontendPageViews`, `SpaceportFrontendBookings`, etc. from generated code — not raw strings |

### 4b. The `useSpan` Hook

Open **frontend/src/hooks/useSpan.ts**.

| Line | What to show | Talking point |
|------|-------------|---------------|
| 12–16 | `useSpan(name, attributes?)` | Reusable React hook. Starts a span on mount, returns `spanRef` and `endSpan()` |
| 22–26 | `useEffect` → `tracer.startSpan(name, ...)` | Span starts when the component mounts |
| 29–31 | Cleanup function: `span.end()` | Safety net on unmount — the span always ends |
| 37–41 | `endSpan()` callback | Call after data loads. Gives the span a meaningful duration instead of "mount → unmount" |

### 4c. Information Desk — All Three Signals

Open **frontend/src/pages/InformationDeskPage.tsx**.

| Line | What to show | Talking point |
|------|-------------|---------------|
| 6–13 | Imports from instrumentation + generated code | Everything typed. `pageViewCounter`, `exhibitViewCounter`, `exhibitDwellHistogram` |
| 19–22 | Span name imports | `SPAN_SPACEPORT_USER_VIEW_INFORMATION_DESK_NAME`, `SPAN_SPACEPORT_EXHIBIT_VIEW_NAME` |
| 33–35 | `useSpan(...)` with exhibit count | **Trace**: page-level span. `spaceport.exhibit.count` = how many exhibits exist |
| 43–47 | `pageViewCounter.add(1, "information_desk")` + `logger.emit(...)` | **Metric** + **Log** on mount. One user action → three signals |
| 82 | IntersectionObserver | `rootMargin: "-45% 0px"` — only a ~10% vertical center strip counts as "visible" |
| 95–107 | Exhibit enters viewport | `tracer.startSpan(SPAN_SPACEPORT_EXHIBIT_VIEW_NAME, { links: [pageSpanCtx] })` — **linked span**, not a child. Shows "related to" without nesting |
| 109–118 | First view of exhibit | Counter bump + span event + structured log — all correlated by trace context |
| 120–128 | Exhibit leaves viewport | Span ends. Dwell histogram records seconds. In Grafana → p50/p95 per exhibit |

> **Demo tip:** Scroll the Information Desk in the browser. Then show Tempo —
> you'll see the page span with linked exhibit spans fanning out.

## 7. The Full Trace (Live in Grafana)

> Make a booking in the browser, then open the trace in Tempo.

Expected trace waterfall:

```
[frontend]  user.place_booking          (INTERNAL)
  └─ [api]  booking.create.server       (SERVER)
       ├─ [api]  pricing.calculate.client   (CLIENT)
       │    └─ [pricing] pricing.calculate.server  (SERVER)
       └─ booking_completed             (EVENT)
```

Point out in the trace:

1. **Span kinds** — INTERNAL → SERVER → CLIENT → SERVER. The CLIENT/SERVER pair creates the service boundary
2. **Attributes** — `spaceport.booking.id`, `spaceport.seat.class`, `spaceport.departure.id` visible on relevant spans
3. **Events** — `booking_completed` with booking ID
4. **Three services, one trace** — `traceparent` header connects them
